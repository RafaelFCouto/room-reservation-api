# Regras de Negócio — room-reservation-api

Este documento descreve as regras de negócio implementadas na API de reserva de salas,
usadas como contexto para avaliação dos comentários gerados pela IA.

---

## RN-01 — Sistema deve impedir reservas com sobreposição de horário na mesma sala

**Jira:** [TCC-5](https://poc-ai-assistent-review.atlassian.net/browse/TCC-5)

Contexto Uma sala não pode ser reservada em um intervalo de tempo que já possui reserva ativa (`PENDING` ou `CONFIRMED`). A verificação deve considerar sobreposição parcial e total, incluindo o caso de borda onde a nova reserva começa exatamente no momento em que outra termina — esse intervalo deve ser permitido. Requisito Atendido RN-01 — Conflito de horário em reservas Cenários Cenário 1: Reserva com sobreposição total    Dado que  a Sala A está reservada das 10h às 12h com status CONFIRMED    Quando  um usuário tenta reservar a Sala A das 10h às 12h    Então  o sistema deve retornar erro 409 com mensagem "Sala indisponível no horário solicitado" Cenário 2: Reserva com sobreposição parcial (início dentro do intervalo existente)    Dado que  a Sala A está reservada das 10h às 12h com status CONFIRMED    Quando  um usuário tenta reservar a Sala A das 11h às 13h    Então  o sistema deve retornar erro 409 com mensagem "Sala indisponível no horário solicitado" Cenário 3: Reserva com sobreposição parcial (fim dentro do intervalo existente)    Dado que  a Sala A está reservada das 10h às 12h com status CONFIRMED    Quando  um usuário tenta reservar a Sala A das 09h às 11h    Então  o sistema deve retornar erro 409 com mensagem "Sala indisponível no horário solicitado" Cenário 4: Reserva adjacente (sem sobreposição — caso de borda)    Dado que  a Sala A está reservada das 10h às 12h com status CONFIRMED    Quando  um usuário tenta reservar a Sala A das 12h às 14h    Então  o sistema deve aceitar a reserva com sucesso (status 201) Cenário 5: Reserva com conflito ignorando status CANCELLED    Dado que  a Sala A possui reserva das 10h às 12h com status CANCELLED    Quando  um usuário tenta reservar a Sala A das 10h às 12h    Então  o sistema deve aceitar a reserva com sucesso (status 201) Critério de aceite: Reservas com `start_time < existing.end_time AND end_time > existing.start_time` devem ser bloqueadas Reservas com `start_time >= existing.end_time` devem ser permitidas (sem sobreposição) Apenas reservas com status `PENDING` ou `CONFIRMED` entram na verificação de conflito Resposta de erro deve indicar claramente o motivo da recusa

## RN-02 — Reserva deve ser criada com antecedência mínima de 24 horas

**Jira:** [TCC-3](https://poc-ai-assistent-review.atlassian.net/browse/TCC-3)

Contexto O sistema deve rejeitar qualquer tentativa de criar uma reserva cujo `start_time` seja inferior a 24 horas a partir do momento da requisição. Isso garante tempo hábil para organização e comunicação entre usuários e gestão do campus. Requisito Atendido RN-02 — Antecedência mínima de 24 horas Cenários Cenário 1: Reserva com menos de 24h de antecedência    Dado que  o horário atual é 2026-06-01T10:00:00Z    Quando  um usuário tenta criar uma reserva com start_time = 2026-06-01T20:00:00Z (10h à frente)    Então  o sistema deve retornar erro 422 com mensagem "Reserva deve ser feita com pelo menos 24 horas de antecedência" Cenário 2: Reserva com exatamente 24h de antecedência    Dado que  o horário atual é 2026-06-01T10:00:00Z    Quando  um usuário tenta criar uma reserva com start_time = 2026-06-02T10:00:00Z (exatamente 24h à frente)    Então  o sistema deve aceitar a reserva com sucesso (status 201) Cenário 3: Reserva com mais de 24h de antecedência    Dado que  o horário atual é 2026-06-01T10:00:00Z    Quando  um usuário tenta criar uma reserva com start_time = 2026-06-05T10:00:00Z (4 dias à frente)    Então  o sistema deve aceitar a reserva com sucesso (status 201) Cenário 4: Reserva para o passado    Dado que  o horário atual é 2026-06-01T10:00:00Z    Quando  um usuário tenta criar uma reserva com start_time = 2026-05-30T10:00:00Z (passado)    Então  o sistema deve retornar erro 422 com mensagem "start_time não pode ser no passado" Critério de aceite: `start_time - now() >= 24 * 60 * 60 * 1000` ms deve ser validado no momento da criação Erro 422 com mensagem descritiva quando a regra for violada A validação deve ocorrer antes da verificação de conflito de horário (RN-01)

## RN-03 — Apenas professores e coordenadores podem reservar salas por mais de 4 horas

**Jira:** [TCC-6](https://poc-ai-assistent-review.atlassian.net/browse/TCC-6)

Contexto Usuários com `profile = student` estão limitados a reservas de no máximo 4 horas contínuas. Usuários com `profile = professor` ou `profile = coordinator` não possuem essa restrição de duração. Requisito Atendido RN-03 — Restrição de duração de reserva por perfil Cenários Cenário 1: Estudante tenta reservar por mais de 4h    Dado que  o usuário autenticado possui profile = student    Quando  ele tenta criar uma reserva com duração de 5 horas    Então  o sistema deve retornar erro 403 com mensagem "Estudantes não podem reservar salas por mais de 4 horas" Cenário 2: Estudante reserva por exatamente 4h    Dado que  o usuário autenticado possui profile = student    Quando  ele tenta criar uma reserva com duração de exatamente 4 horas    Então  o sistema deve aceitar a reserva com sucesso (status 201) Cenário 3: Professor reserva por mais de 4h    Dado que  o usuário autenticado possui profile = professor    Quando  ele tenta criar uma reserva com duração de 8 horas    Então  o sistema deve aceitar a reserva com sucesso (status 201) Cenário 4: Coordenador reserva por mais de 4h   Dado que  o usuário autenticado possui profile = coordinator    Quando  ele tenta criar uma reserva com duração de 6 horas    Então  o sistema deve aceitar a reserva com sucesso (status 201) Critério de aceite: Duração calculada como `end_time - start_time` Limite de 4h = 14400000 ms — deve ser definido como constante nomeada no código (`MAX_STUDENT_DURATION_MS`), não como magic number Verificação deve consultar `user.profile` do usuário autenticado, não de parâmetro da requisição Erro 403 para violação de permissão por perfil

## RN-05 — Cancelamento de reserva só é permitido quando status = PENDING

**Jira:** [TCC-8](https://poc-ai-assistent-review.atlassian.net/browse/TCC-8)

Contexto Somente reservas com status `PENDING` podem ser canceladas. Tentativas de cancelar reservas com status `CONFIRMED` ou `CANCELLED` devem ser rejeitadas. Ao cancelar, o slot de horário deve ser liberado imediatamente para novas reservas. Requisito Atendido RN-05 — Cancelamento restrito a reservas pendentes Cenários Cenário 1: Cancelar reserva com status PENDING    Dado que  existe uma reserva com status PENDING    Quando  o proprietário da reserva envia DELETE /reservations/:id    Então  o sistema deve atualizar o status para CANCELLED (200)    E  o slot de horário deve ser liberado para novas reservas Cenário 2: Tentar cancelar reserva CONFIRMED    Dado que  existe uma reserva com status CONFIRMED    Quando  o proprietário da reserva envia DELETE /reservations/:id    Então  o sistema deve retornar erro 422 com mensagem "Apenas reservas PENDING podem ser canceladas" Cenário 3: Tentar cancelar reserva já CANCELLED    Dado que  existe uma reserva com status CANCELLED    Quando  o proprietário da reserva envia DELETE /reservations/:id    Então  o sistema deve retornar erro 422 com mensagem "Reserva já foi cancelada" Cenário 4: Cancelamento libera slot para nova reserva    Dado que  a Sala A possui reserva das 10h às 12h com status PENDING    Quando  o proprietário cancela essa reserva    E  outro usuário tenta reservar a Sala A das 10h às 12h    Então  a nova reserva deve ser criada com sucesso (status 201) Critério de aceite: Validação de status deve ocorrer antes de qualquer operação de escrita Transição permitida: `PENDING → CANCELLED` Transições proibidas: `CONFIRMED → CANCELLED`, `CANCELLED → CANCELLED` Liberação do slot deve ser atômica com a operação de cancelamento Erros de transição inválida retornam 422

## RN-06 — Sistema deve bloquear reservas e listagem de salas com status MAINTENANCE

**Jira:** [TCC-9](https://poc-ai-assistent-review.atlassian.net/browse/TCC-9)

Contexto Salas com `status = MAINTENANCE` devem ser completamente excluídas do fluxo de reservas: não podem ser reservadas, não devem aparecer em listagens de salas disponíveis e não podem ter novas reservas confirmadas por qualquer tipo de usuário, incluindo coordenadores. Requisito Atendido RN-06 — Salas em manutenção não podem ser reservadas Cenários Cenário 1: Tentar reservar sala em manutenção    Dado que  a Sala B possui status = MAINTENANCE    Quando  qualquer usuário tenta criar uma reserva para a Sala B    Então  o sistema deve retornar erro 422 com mensagem "Sala indisponível: em manutenção" Cenário 2: Sala em manutenção não aparece na listagem de disponíveis    Dado que  existem 3 salas: Sala A (AVAILABLE), Sala B (MAINTENANCE), Sala C (AVAILABLE)    Quando  um usuário consulta GET /rooms/available    Então  a resposta deve conter apenas Sala A e Sala C    E  Sala B não deve aparecer na listagem Cenário 3: Coordenador não pode confirmar reserva de sala em manutenção    Dado que  a Sala B possui status = MAINTENANCE    E  existe uma reserva PENDING para a Sala B (criada antes da mudança de status)    Quando  um coordenador tenta atualizar o status da reserva para CONFIRMED    Então  o sistema deve retornar erro 422 com mensagem "Sala indisponível: em manutenção" Cenário 4: Sala criada com status MAINTENANCE não fica disponível para reservas    Dado que  um coordenador cria uma nova sala com status = MAINTENANCE    Quando  qualquer usuário tenta reservar essa sala imediatamente após a criação    Então  o sistema deve retornar erro 422 com mensagem "Sala indisponível: em manutenção" Critério de aceite: Verificação de `room.status` obrigatória em todas as operações de criação e confirmação de reserva Endpoint `GET /rooms/available` deve filtrar salas com `status != MAINTENANCE` diretamente na query do banco, não em memória A restrição se aplica a todos os perfis de usuário, sem exceções Erro 422 com mensagem indicando o motivo

## RN-07 — Apenas professores e coordenadores podem cadastrar novas salas

**Jira:** [TCC-10](https://poc-ai-assistent-review.atlassian.net/browse/TCC-10)

Contexto O endpoint de criação de salas (`POST /rooms`) deve verificar o perfil do usuário autenticado antes de processar a requisição. Usuários com `profile = student` devem ser bloqueados com erro de autorização, independentemente dos dados enviados. Requisito Atendido RN-07 — Cadastro de salas restrito a professores e coordenadores Cenários Cenário 1: Estudante tenta cadastrar sala    Dado que  o usuário autenticado possui profile = student    Quando  ele envia POST /rooms com dados válidos de uma sala    Então  o sistema deve retornar erro 403 com mensagem "Permissão negada: apenas professores e coordenadores podem cadastrar salas" Cenário 2: Professor cadastra sala com sucesso    Dado que  o usuário autenticado possui profile = professor    Quando  ele envia POST /rooms com dados válidos (name, capacity, building)    Então  o sistema deve criar a sala e retornar status 201 com os dados da sala criada Cenário 3: Coordenador cadastra sala com sucesso    Dado que  o usuário autenticado possui profile = coordinator    Quando  ele envia POST /rooms com dados válidos    Então  o sistema deve criar a sala e retornar status 201 com os dados da sala criada Cenário 4: Cadastro com capacity inválida    Dado que  o usuário autenticado possui profile = professor    Quando  ele envia POST /rooms com capacity = 0 ou capacity = -5    Então  o sistema deve retornar erro 422 com mensagem "Capacidade deve ser um número inteiro positivo" Cenário 5: Requisição sem autenticação    Dado que  a requisição não possui token de autenticação    Quando  ela é enviada para POST /rooms    Então  o sistema deve retornar erro 401 antes de verificar o perfil Critério de aceite: Verificação de `user.profile` deve ocorrer via middleware de autorização, não dentro da função de controller Profiles autorizados: `professor` e `coordinator` `capacity` deve ser validada como inteiro positivo (`> 0`) Erro 403 para perfil não autorizado, 401 para não autenticado Campos obrigatórios: `name`, `capacity`, `building` — validação não deve ser duplicada entre middleware e controller
